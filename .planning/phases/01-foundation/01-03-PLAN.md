<objective>
Scaffold the React Native frontend so a developer can start building screens in Phase 3 without any further setup work. This plan creates `/frontend/` from scratch using Expo SDK 55, installs every library Phase 2-5 will need (navigation, server-state, client-state, fonts, secure storage, performance list, forms, HTTP, bottom sheet), wires up a typed swappable theme (translated from `design/high_performance_dark_mode/DESIGN.md`), builds the 3-tab bottom navigator with native-stack per tab, and stands up the auth store + axios client with a Bearer-token interceptor.

Purpose: D-16 requires components to call `useTheme()` — never raw hex. D-17, D-18 lock the contract that swapping themes is one line. D-19 holds the splash until Space Grotesk fonts load. D-21 makes OTP mockable. The cumulative effect: Phase 2 only writes auth screens, Phase 3 only writes the listing form — neither phase debugs setup. The cost of getting this wrong now is amplified across 4 phases.

- A `/frontend/` Expo project that runs `npx expo start` cleanly with all 3 tab screens rendering themed text
- A typed theme system whose dark variant uses the EXACT hex values from DESIGN.md YAML frontmatter
- An auth store backed by `expo-secure-store` (Keychain/Keystore) — never AsyncStorage
- An `apiClient` whose request interceptor reads `useAuthStore.getState().token` and attaches `Authorization: Bearer <token>` automatically
- TypeScript that compiles clean (`npx tsc --noEmit`)
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile JS bundle -> device storage | JWT tokens cross here. Must use `expo-secure-store` (Keychain/Keystore), NEVER `AsyncStorage` (unencrypted). |
| Mobile JS bundle -> public env | `EXPO_PUBLIC_*` env vars are inlined into the bundle at build time. Anything in those vars is public. The Cloudinary `api_secret` and the MSG91 `authkey` MUST NOT live in `EXPO_PUBLIC_*` vars. |
| Mobile -> Backend API | All API calls cross the public internet. The Bearer token is the only thing identifying the user — must be sent only over HTTPS in production (Railway provides HTTPS by default). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-01 | Information Disclosure | JWT stored in AsyncStorage | mitigate | Auth store uses `expo-secure-store` exclusively (Keychain on iOS, EncryptedSharedPreferences on Android). The plan explicitly forbids `AsyncStorage` for tokens (per CLAUDE.md "What NOT to Use"). |
| T-01-03-02 | Information Disclosure | Cloudinary `api_secret` shipped in mobile bundle | mitigate | `frontend/.env.example` documents only `EXPO_PUBLIC_API_URL`. NO `EXPO_PUBLIC_CLOUDINARY_*` keys are used. The mobile app obtains a short-lived signature from `/api/v1/upload/signature` (backend holds the secret). |
| T-01-03-03 | Tampering | API URL hijack via build-time env injection | accept | `EXPO_PUBLIC_API_URL` is build-time injected. A malicious build can substitute it, but that's a supply-chain concern outside Phase 1 scope. |
| T-01-03-04 | Repudiation | Token persists after logout if SecureStore.deleteItemAsync fails silently | mitigate | `clearToken` awaits `deleteItemAsync` and only updates Zustand state after success. Errors propagate so callers see the failure. |
| T-01-03-05 | Spoofing | Bearer token sent on requests after logout | mitigate | The interceptor reads `useAuthStore.getState().token` synchronously per request — once `clearToken` resolves, the token field is null and the next request sends no Authorization header. |
</threat_model>

```typescript
import { create } from 'zustand';
import * as SecureStore from 'expo-secure-store';

const TOKEN_KEY = 'revvbase_auth_token';

interface AuthState {
  token: string | null;
  isAuthenticated: boolean;
  setToken:   (token: string) => Promise<void>;
  clearToken: ()              => Promise<void>;
  loadToken:  ()              => Promise<void>;
}

export const useAuthStore = create<AuthState>((set) => ({
  token: null,
  isAuthenticated: false,
  setToken: async (token: string) => {
    await SecureStore.setItemAsync(TOKEN_KEY, token);
    set({ token, isAuthenticated: true });
  },
  clearToken: async () => {
    await SecureStore.deleteItemAsync(TOKEN_KEY);
    set({ token: null, isAuthenticated: false });
  },
  loadToken: async () => {
    const token = await SecureStore.getItemAsync(TOKEN_KEY);
    set({ token, isAuthenticated: !!token });
  },
}));
```
